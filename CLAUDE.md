# プロジェクト概要

ボードゲーム「7 SHADES OF S:EVEN」リメイク（仮）のデジタル版アプリ。
姉妹プロジェクトとして、隣接フォルダ（親フォルダ直下）に「7 SHADES OF S:EVEN 戦績管理システム」
（プレイテスト結果の記録用アプリ、index.html）が別リポジトリとして存在するが、本プロジェクトとは無関係。

## ゴールと段階

- 最終ベンチマークは MTGA（Magic: The Gathering Arena）＝カード効果を完全に自動処理する形。
- ただしいきなりそこは目指さず、まずはユドンリウムコネクト（ユドコネ）のような
  「アナログコンポーネントを画面に移植しただけ」の形を目指す。カード効果はテキスト表示のみで、
  処理はプレイヤーの自己申告に委ねる。
- ルールブックやカード効果は今後変更される可能性がある前提で設計する。

## 対戦形態（現状の合意事項）

- 最終的にはオンライン非同期対戦（同時接続ではなく、各プレイヤーが自分のタイミングで手番を進める形）を目指す。
- 実装は1人用（対戦相手なしのテストプレイ用）から始める。
- ただし「1人用→オンライン非同期」の移行で手戻りが出ないよう、最初から下記のアーキテクチャ方針を徹底する。

## アーキテクチャ方針

- ゲーム状態（盤面・手札・山札・フェーズなど）を1つのシリアライズ可能なオブジェクト（JSON）として持つ。
- 状態変更は全て「アクション」として表現し、`現在の状態 + アクション → 次の状態` の純粋な形にする
  （Reduxライクな設計）。1人用の段階ではアクションをローカルで即時適用するだけにしておき、
  オンライン非同期化する際は同じアクションをバックエンドに保存して同期する流れにそのまま乗せる。
- カード効果・カードデータはコードに埋め込まず、外部データ（JSON等）として持つ（変更に強くするため）。

## 技術スタック（現状の合意事項）

- 姉妹プロジェクト（戦績管理システム）に倣い、ビルド不要の単一HTML＋Tailwind CSS（CDN）＋Supabase
  （オンライン非同期対戦のバックエンド）という軽量構成を踏襲する。
- プラットフォームはデスクトップアプリを最終形としたいが、まずはブラウザで動くものを作り、
  将来的にTauri等でラップしてデスクトップアプリ化する想定。スマホとPCのクロスプレイも目標。

## Phase 1 実装状況

- 方針: CSS疑似3D（`perspective`+`rotateX`）で開始。3D化が必要になった場合は描画層（`src/main.js`の描画処理）だけを差し替える想定で、ゲームロジックとは分離して実装する。
- モバイル対応: PC（横長ブラウザ）優先。スマホ横向きは意識するが、実機での細かい調整はPhase1完了後に回す（現状、縦幅が狭い画面では上下のプレイヤーゾーンが見切れる）。
- ローカル確認用サーバー: ルートの `.claude/launch.json` に `7shades-digital` という設定名で追加済み（`npx serve 7shades-digital -l 3001`）。ルート直下の`static-serve`(3000番)は姉妹プロジェクト用なのでポートを分けている。
- 静的レイアウト（盤面7x7・4人分のロックエリア/手札ゾーン・共有山札等）を [index.html](index.html) / [src/style.css](src/style.css) / [src/main.js](src/main.js) にダミーデータで実装済み。まだゲーム状態（GameState）とは未接続。
- **注意（ハマりどころ）**: CSS疑似3Dの箱に `class="table"` を付けたところ、Tailwind CDNの組み込みユーティリティクラス`table`（`display:table`）と衝突し、`display:grid`が上書きされるバグが発生した。Tailwind使用時はカスタムクラス名がTailwindの既存ユーティリティ名（`table`, `hidden`, `flex`, `grid`, `block`, `container`等）と衝突しないよう注意する（今回は`game-table`に変更して回避）。
- 座席配置: 手前(南)=A(自分)、左(西)=B、奥(北)=C、右(東)=D。盤面を上から見て時計回り(A→B→C→D)がルールブックの「時計回り」ターン順とそのまま一致するよう統一した。
- 手札は「テーブルに置く」のではなく「持ち上げて扇状に持っている」見た目。自分(A)は`rotateX(-40deg)+translateZ`で大きく持ち上げ扇状に開く（上下方向）。他プレイヤー(B/C/D)は逆に`rotateX(18deg)`で盤面側へ伏せ、裏向き柄（`.is-facedown`）にして密集させる（MTGA参考画像の対面プレイヤー表現に合わせた）。全員分カードごとに個別回転＋位置をJS側で計算する扇形式（`layoutFan`参照）。左右(B/D)も上下(A/C)と同じ考え方で縦方向に扇状に開く（扇コンテナ全体を90度回転させる旧方式はカード自体が横倒しになるバグがあったため、個々のカードにのみ回転をかける方式にした）。弧の向き（外側のカードが盤面から逃げる方向）は`ARC_SIGN`で座席ごとに定義。
- 手札・駒・山札系のカードはすべて正方形（このゲームの物理カードが正方形のため）。
- **駒の見た目は何度か作り直した**（試行錯誤の記録）:
  1. 3面をpreserve-3dで組む真の3D立方体 → 盤面のrotateX(42deg)との兼ね合いで正面以外がほぼ潰れる。打ち消し回転（ビルボード化）も試したが逆に正面が潰れて失敗。
  2. 2DのskewX/skewYで組む疑似アイソメトリック立方体（top/front/sideの3枚）→ 見た目は良くなったが、駒がセルより大きくはみ出すため「DOM順で後に来る隣のセル」に隠され、脚だけ残る崩れ方をした（`.cell`同士が同じスタッキング文脈でDOM順に描画されるため）。駒のあるセルに`z-index`を与えて一旦解決したが、複数面構成自体が壊れやすかった。
  3. ユドナリウム(TK11235/udonarium、Angular+TypeScript製、three.js等は不使用でCSSベース)の`game-character`コンポーネントを調査し、非対称な太さ・濃淡のボーダーだけで立体感を出す手法を試した（`.pedestal-grab-border`参考）。頑丈だが、ユーザーから「駒はユドナリウムでいう『地形』に該当する」と指摘があり、参照コンポーネントが違うと判明。
  4. **`terrain`（地形）コンポーネントを調査し、その技法をそのまま移植**。「床」を`translateZ(height)`で持ち上げ、4枚の壁を角ごとに異なる`transform-origin`+`rotateZ`/`rotateX`の組み合わせで側面に貼り付ける、正しい組み方のpreserve-3d立方体（`buildCubePiece`参照、CSSクラスは`.piece-top`/`.piece-wall-front`/`.piece-wall-back`/`.piece-wall-left`/`.piece-wall-right`）。移植直後、壁の高さが常にゼロに潰れる新たなバグが発生したが、原因は**`.piece`に付けていた`filter: drop-shadow()`**だった。CSSの`filter`は要素とその子孫を強制的に2D平坦化する仕様があり、`transform-style: preserve-3d`を無効化してしまう。影を`box-shadow`に変更して解決。この技法なら、駒がどの位置にあってもどの面がどれだけ見えるかはブラウザの3D計算が自動でやってくれるため、以前のようなセル位置に応じたJS側の手動調整（leanFactor）は不要になった。
  - 教訓1: 見た目を真似たい対象（ユドコネ）が実際どう実装されているかをオープンソースで確認する時は、コンポーネント名（駒＝character/terrainのどちらか）を早い段階でユーザーに確認する。
  - 教訓2: `transform-style: preserve-3d`を使う要素には`filter`（drop-shadow等）を付けない。影が欲しい場合は`box-shadow`を使う。
  - 教訓3: `backface-visibility: hidden`もユドナリウム本家では設定されていた（移植時に見落としていた）。真の3D構成を組む際は本家CSSを可能な限り忠実に移植する。
  - 教訓4: 強めの`perspective`（近いカメラ）の下では、駒の上面（カメラに近い＝拡大率が高い）と側面（奥行きがあり拡大率が異なる）の間にわずかな継ぎ目の隙間ができることがある。側面の高さを実際の駒の高さより少し（0.15rem程度）大きくして上面と重ねることで隙間を隠せる。
- 自分の手札カード（`.hand-card.is-self`）は視認性のため他プレイヤーの手札より大きく表示している。それに合わせて`.game-table`のgrid-template-rows（手前の行のみ拡大）、`.zone-bottom .hand-area`の高さ、扇の間隔(layoutFanのspacing)も調整済み。
- 盤面の奥行き感（台形っぽさ）は、`perspective`の値を小さくする（=カメラを近づける）ことで自然に強調できる。手動で盤面を台形に変形するのではなく、既存の3D構成（`perspective`+`rotateX`）のパラメータ調整で対応した（`perspective: 2200px→1150px`）。
- 山札は左上、エターナルカードは左下、捨て場は右下（山札と対角）に配置済み。
- 相手（B/C/D）の手札の弧（外側のカードを盤面から遠ざける演出）は、3D変形と組み合わさると非対称に見える不具合が出たため撤去し、回転のみのシンプルな扇にした（自分の手札は弧を維持）。
- 山札・エターナルカード・捨て場は、枚数に応じて厚みを持つ「積み重ね」表示にした（`.stack`/`.stack-top`/`.stack-front`、`buildCardStack`参照）。駒(`.piece`)と同じ「床+壁」の考え方だが面は上面と正面の2枚のみ。1枚あたり0.6px・最低2.4pxの厚み。初期山札は63枚（通常カード112枚－盤面49枚）、エターナルカードは7枚、捨て場は0枚。将来、盤面マスのカードスタックにも同じ`buildCardStack`を流用する想定。
- ウィンドウサイズが変わっても手札等が見切れないよう、`fitTableToViewport()`で`.game-table`の`scale`をJS側から動的に調整するようにした（`--table-tilt`というCSS変数でrotateX角度をCSS/JS間で共有し、scaleだけを画面サイズに応じて計算）。`resize`イベントでも再計算する。最大1.15倍まで拡大、それ以上大きい画面では頭打ちにしている。
- ローカルネットワーク内の実機（iPhone等）から確認する場合は `http://localhost:3001` ではなくPCのLAN IPアドレス（例: `http://192.168.10.137:3001`、環境により変わる）でアクセスする必要がある。`localhost`は「そのデバイス自身」を指すため他端末からは繋がらない。
- プレイヤーBの手札は`rotate(180deg)`で回転させている（`.zone-left .hand-fan.is-opponent`）。
- `.zone`（手札エリア・山札エリア等を囲っていた四角い枠）の背景・ボーダー・box-shadowは削除し、透明なレイアウト用コンテナのみにした。
- カラーパレットは実際のゲームの色番号に更新済み: 赤`#c70025` 橙`#ee781f` 黄`#fabe00` 緑`#22ac38` 青`#1bb8ce` 桃`#f19ec2` 紫`#915da3`（`:root`の`--color-*`参照）。
- **実物のプレイマット画像を背景に使用**: `画像素材/プレイマット.png`（1080x1080、gitignore対象＝ローカルのみ）を`assets/playmat.png`にコピーして`.arena`の`background-image`に設定（gitで管理、こちらはコミット対象）。画像はパレット形式(indexed PNG)でPythonが使えない環境だったため、Node.jsの`zlib`でPNGを自前デコードして構造を解析しようとしたが、画像内の細かい模様・陰影のせいで自動検出は不確実だった。最終的に「画像全体をアリーナ全体に等倍（`background-size:100% 100%`、`background-position:0 0`）でマッピングし、`--lock-thickness`（ロック帯の太さ）をブラウザで実測しながら調整する」という目視確認ベースの方法で十分な精度の位置合わせができた。3D変形（rotateX）がかかっていると比較しづらいので、位置合わせ中は`game-table`の`transform`を`none`に一時的に戻して平面表示で確認すると良い。
  - 教訓5: 複雑なテクスチャを含む画像のレイアウト解析は、ピクセル解析より「実際にブラウザで描画して目視確認・調整」の方が早い場合がある。特に3D変形がかかった状態では正確な比較ができないので、位置合わせ時は一時的に平面化する。
- **管理者モード（[src/admin.js](src/admin.js)）を追加**: 画面左上の「⚙ 管理者モード」ボタンから、プレイマット画像の位置合わせ用パラメータ（`--playmat-size-x/y`, `--playmat-pos-x/y`, `--lock-thickness`, `--arena-gap`）をスライダーでその場調整できるようにした。ユーザーが微調整して「出力をコピー」した`:root { ... }`のテキストをもらったら、その値を[src/style.css](src/style.css)の対応する`:root`変数にそのまま反映する運用。開発用ツールなので常時表示可能にしてあり、ゲーム本編のUIではない（将来的に本番ビルドでは非表示にするか検討）。
